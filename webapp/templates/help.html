Help & Documentation

Learn how the DrugCLIP virtual screening workflow operates, how to define binding sites, and which screening mode to choose for your library.

DrugCLIP performs virtual screening by encoding both the protein binding pocket and candidate molecules into a shared embedding space, then ranking compounds by similarity. The pipeline runs in four sequential steps:

1 Pocket Extraction

The binding pocket is extracted from your protein structure (PDB file) using the binding site definition you provide. Residues within the specified cutoff radius are identified and converted to LMDB format for the model.

2 Library Conversion

Your compound library (SDF, SMI, or SMILES file) is converted into LMDB format. Each molecule is parsed and stored with its 3D conformer so the model can process it efficiently on GPU.

3 Encoding

Both the pocket and all molecules are encoded into fixed-length embedding vectors using the DrugCLIP neural network. This step runs on GPU and produces compact representations that capture binding-relevant features.

4 Scoring

Similarity scores are computed between the pocket embedding and every molecule embedding. Compounds are ranked by score, and the top fraction (e.g., top 2%) is returned as hits in the results file.

You must define where on the protein the binding pocket is located. DrugCLIP supports four methods — choose the one that matches the structural information you have available.

Co-crystallized Ligand

Upload a file containing the coordinates of a ligand that is already bound in the crystal structure. The pocket is defined as all protein residues within the cutoff distance of this ligand.

When to use: You have a PDB structure with a known ligand bound in the active site (e.g., from a co-crystal structure in the PDB). This is the most common and reliable method.
HETATM Residue Name

Specify the 3-letter residue name of a heteroatom (non-protein molecule) present in the PDB file. The tool will locate that molecule and define the pocket around it.

When to use: You know the 3-letter code of a bound molecule in your PDB file (e.g., "JHN", "ATP", "NAG") but don't have a separate ligand file. Check the HETATM records in your PDB to find the code.
Explicit XYZ Coordinates

Provide the X, Y, and Z coordinates (in Ångströms) of the center of the binding site. All residues within the cutoff radius of this point will form the pocket.

When to use: You know the center of the binding site from literature, a docking study, or visual inspection in a molecular viewer (e.g., PyMOL, ChimeraX). Useful when no ligand is present in the structure.
Protein Residue Numbers

Specify the residue numbers that form the binding pocket directly. You can optionally provide a chain ID if your protein has multiple chains.

When to use: You know which residues form the binding pocket from mutagenesis data, literature, or sequence analysis. This gives you the most precise control over pocket definition.

DrugCLIP offers two screening modes to handle libraries of different sizes efficiently on the HPC cluster.

Standard Mode

Runs the entire screening pipeline as a single SLURM job. All four steps (pocket extraction, library conversion, encoding, scoring) execute sequentially on one GPU node.

Best for:
  • Libraries up to approximately 1 million compounds
  • Quick screening runs
  • Testing and validation with smaller datasets
Configuration:

No additional parameters needed beyond the basic form fields.

Large-Scale Mode

Splits the compound library into chunks and processes them in parallel across multiple GPU nodes. Results are merged at the end.

Best for:
  • Libraries with more than 1 million compounds
  • Large commercial libraries (e.g., Enamine REAL, ZINC)
  • When faster turnaround is needed for big datasets
Configuration:
  • Chunk size — number of compounds per chunk (default: 1,000,000)
  • Partition — SLURM partition to use (default: ga100)
  • Max parallel jobs — maximum concurrent SLURM jobs (default: 50)

My job has been PENDING for a long time. Is something wrong?

A PENDING status means your job is waiting in the SLURM queue for a GPU node to become available. During busy periods, wait times can range from minutes to hours depending on cluster load. Check the job log for any scheduler messages. If the job stays PENDING for more than a few hours, the partition may be fully occupied — try again later or contact your HPC administrator.

My job FAILED. How do I find out what went wrong?

Go to the job detail page and check the log viewer section. The last 50 lines of the SLURM log are displayed automatically for failed jobs. Common causes include: invalid PDB format, empty compound library, insufficient GPU memory for very large pockets, or missing atoms in the structure. Fix the input and resubmit.

What file formats are supported for the compound library?

The compound library can be uploaded in the following formats:

  • .sdf — Structure-Data File with 3D coordinates
  • .smi — SMILES file (one SMILES per line)
  • .smiles — Same as .smi
  • .txt — Plain text file with one SMILES per line

For SMILES-based formats, 3D conformers are generated automatically during the library conversion step.

How do I choose the right cutoff radius?

The default cutoff of 10.0 Å works well for most drug-sized binding pockets. If your target has a particularly large or shallow binding site, you may want to increase it to 12–15 Å. For very small or deeply buried pockets, 8 Å may be sufficient. When in doubt, stick with the default.

Can I screen the same protein against multiple libraries?

Yes. Submit a separate job for each library using the same PDB file and binding site definition. Each job runs independently and produces its own results file. You can compare hits across libraries by downloading and merging the results.
